Starting with NTL

This page is the main reference for the language. It starts simple and keeps going until it covers the bits people usually need in real projects: syntax, commands, modules, stdlib, runtime notes, and the rough edges that are worth knowing about early.

NTL is still moving fast. Some details may shift over time, and that is normal for a project built by one person. The goal here is clarity first, not pretending the language is finished forever.

Starting

The shortest way to understand NTL is to look at a tiny file, run it, and then add one thing at a time. NTL is designed to feel direct. You import only what you need, keep the program structure small, and move on. 

val io = @import("std.io")

fn main() {
  io.log("Hello from NTL")
}

That example shows the usual pattern: import a module, define a function, print something, and build from there. It is simple on purpose.

Installation

Use the official installer for your platform. The download page mirrors the release assets from the GitHub releases folder and keeps the list clear.

Linux

amd64 and arm64 builds are available. Most users only need the matching binary and the usual execute permission.

macOS

Intel and Apple Silicon builds are both listed separately so people do not have to guess.

Android / Termux

The Android build is there for Termux users and other ARM64 environments on mobile devices.

Windows

Windows gets its own installer asset and a simple setup path for users who prefer the .exe release.

A quick note: because NTL is built by one person, installer names and packaging details can change. The docs try to stay honest about that instead of pretending everything is perfectly static.

Basic syntax

NTL keeps syntax compact. It borrows the useful feeling of scripting languages, but it does not try to be a clone of anything else.

val io = @import("std.io")

val name = "Alice"
var count = 1

fn main() {
  io.log(name, count)
}
  • val creates an immutable binding.
  • var creates a mutable binding.
  • fn defines a function.
  • Modules are imported explicitly with @import.

Variables and values

NTL uses values that are easy to read in examples and easy to pass around in real code. The reference docs in the source tree show strings, numbers, arrays, objects, booleans, and null-like values as the day-to-day building blocks.

Immutable bindings

val is the safe default. Use it when the value should not be reassigned.

Mutable bindings

var is there when the variable really needs to move. Keep it visible and intentional.

val title = "NTL"
var retries = 0
retries += 1

The language is still documented as dynamically typed in the current tree, but the compiler and runtime still care about consistency and clear runtime behaviour.

Control flow

Conditionals and loops are kept straightforward. The goal is to keep the code readable when you come back to it later.

val io = @import("std.io")

fn main() {
  var i = 0
  while i < 5 {
    if i % 2 == 0 {
      io.log("even", i)
    } else {
      io.log("odd", i)
    }
    i += 1
  }
}

For pattern-heavy code, the language tree also includes pattern matching and asynchronous examples, which are documented in the source package.

Functions

Functions are the place where most of the language style shows up. Keep them small, name them clearly, and use the last expression style where the current runtime expects it. 

val io = @import("std.io")

fn greet(name) {
  "Hello, " + name + "!"
}

fn main() {
  io.log(greet("David"))
}

The source docs lean into a simple rule: write a function for one job, make the result easy to follow, and let the runtime do the rest.

Modules

NTL uses explicit imports. Nothing is magically present unless you import it. That makes the code a bit more honest and a lot easier to reason about in larger projects. 

val io = @import("std.io")
val fs = @import("std.fs")
val http = @import("std.http")

The package system in the source tree also supports GitHub-based module installs and subfolder installs, which is handy when a repo holds more than one module.

CLI commands

The command-line tool is the quickest way to move from source to output. The exact backend depends on your target and configuration, but the surface area stays friendly.

Standard library

NTL ships with a broad standard library. The source tree shows modules for I/O, filesystem, HTTP, OS access, crypto, environment variables, databases, validation, XML, websockets, PDF, mail, JWT, OAuth2, PostgreSQL, MySQL, Redis, RabbitMQ, Stripe, Excel, GraphQL, and more.

io — console output and input

The I/O module is the one you meet first. It handles logging, warnings, errors, and other terminal output helpers. 

val io = @import("std.io")

fn main() {
  io.log("Hello, World")
  io.success("Build complete")
  io.warn("This is just a warning")
  io.error("Something went wrong")
}

It is the module most people use while learning, which makes it a good place for small tests and output checks.

fs — filesystem

The filesystem module handles reading, writing, appending, checking paths, copying, and watching files. That makes it a natural fit for build tools and small automation tasks. 

val fs = @import("std.fs")
val io = @import("std.io")

val text = fs.readFile("notes.txt")
fs.writeFile("out.txt", text)
if fs.exists("out.txt") {
  io.log("saved")
}

http — HTTP client and server

HTTP is one of the modules that makes NTL feel more like a real toolchain than a toy language. The source docs include client calls, request objects, and server examples. 

val http = @import("std.http")
val io   = @import("std.io")

fn main() {
  val res = http.get("https://example.com")
  io.log(res.status)
}

os — system interaction

The OS module gives you process execution, environment variables, path handling, and basic system calls in a way that suits scripting and tooling. 

val os = @import("std.os")
val io = @import("std.io")

fn main() {
  val result = os.exec("git status")
  io.log(result.stdout)
}

alloc — low-level buffers

The alloc module is where NTL gets closer to raw memory work. It is useful when you need bytes, slices, or a buffer-oriented view of data. 

val alloc = @import("std.alloc")
val io    = @import("std.io")

val buf = alloc.buffer(16)
buf.writeString(0, "hello")
io.log(buf.size())
buf.free()
Low-level modules are powerful, but they also deserve respect. The docs should be honest about the fact that raw buffers can crash things if they are misused.

Architecture

The architecture section in the source tree explains the split between the compiler and the runtime. In plain English: Go handles the front half, Zig handles the runtime side, and the release model aims to keep the toolchain practical.

Compiler side

The compiler parses source, builds bytecode or a native path, and coordinates the build flow.

Runtime side

The runtime and JIT live in Zig and are designed to keep execution fast and portable.

Extensions

Extensions are one of the nicest parts of the project. They let the runtime grow without turning every feature into a giant rewrite. The source docs show how to register native functions and how the build system discovers them. 

val io = @import("std.io")

fn square(x) {
  x * x
}

fn main() {
  io.log(square(12))
}

Zig gets a lot of credit here. It is a very good fit for the runtime and for the kind of low-level control NTL needs. That part is worth saying plainly.

Troubleshooting

  • Installer fails: check the file name and make sure you grabbed the right asset for your platform.
  • Command not found: confirm the binary is in your PATH or call it from the install location directly.
  • Module not found: verify the import name and package path.
  • Weird runtime behavior: this can happen while the language is moving quickly. Re-test on the latest build and report the issue if it still shows up.

FAQ

Is NTL finished?

No. It is a growing project. The docs should say that straight up.

Can the docs be expanded later?

Yes. That is the point of the structure. More modules, more examples, more pages, no drama.
